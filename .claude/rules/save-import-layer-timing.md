# Save-import: Layer A is pre-load XML-only (no `Game1`); the durable bind is Layer B at SaveLoaded

The `saves import` feature splits into two layers with **opposite engine availability**, and conflating them breaks the operator path:

- **Layer A — the XML transform** (clone-blank a "Server" master from the imported save's own `<player>`, reparent the owner into `<farmhands>`, stamp `userID`) runs in the **console command, pre-load at the title screen, with no game loaded**. `Game1.getFarm()` / any live `Farmer` is unavailable. So Layer A must be pure `System.Xml` — it cannot clone a blank master from a live engine farmer, and the blank "Server" `<player>` must be sourced from XML (clone-and-blank the staged file's own `<player>`, which is present pre-load). Operate on a **staged copy**, never the original.
- **Layer B — the load-time finalizer** runs from `CabinManagerService.OnSaveLoaded` with a **fully live game**. This is why the durable steps live here: build a cabin, `Cabin.AssignFarmhand(owner)` (homes the farmhand), then re-stamp `userID.Value`. A pure-XML userID stamp does **not** survive load for the headline swap case — the demoted owner is cabin-less at `ResetFarmhandState` time, so `TryAssignFarmhandHome` returns false and vanilla clears the userID mid-load (`NetWorldState.cs:760-774`). The re-stamp must come *after* a live `AssignFarmhand` homes it.

The trap: the `/test/import_save` endpoint runs `ExecuteImport` inside `RunOnGameThreadAsync` (mirroring `/test/stamp_claim`, `ApiService.TestEndpoints.cs:556-628`) — so the **test path has a live engine the operator path does not**. Don't let that leak into Layer A's design.

**Why:** During plan review I nearly recommended "engine-clone the blank master" as a cleaner alternative to an embedded authored template — invalid, because Layer A has no `Game1` in the operator console path. The XML-only constraint is real, not unnecessary brittleness. The chosen approach (clone the imported save's own `<player>` and blank it) keeps it XML-only while matching the save's exact game-version `Farmer` schema (no frozen template to drift).

**How to apply:** Any Layer A logic must be pure `XmlDocument` with no `Game1`/SMAPI-game dependency; engine-dependent steps (cabin build, farmhand home, userID re-stamp) belong in Layer B. The Layer B finalizer must run as the **first statement of `OnSaveLoaded`, before `Data.Read()`** and the reconciliation chain (`SyncExistingCabins` → `ClearStaleFarmhandReferences` → `ClearAbandonedCabinClaimsOnLoad` → `EnsureAtLeastXCabins`), with the owner `isCustomized=true`, so the stale-ref and abandoned-claim sweeps don't strip the fresh home/userID (see [[cabin-system]] invariant 8). Reuse `BuildNewCabin`/`BuildNewCabinVisible` (the None-strategy placement path, [[cabin-system]] invariant 9) — but note they return `bool` and discard the `Building`, and `buildStructure` auto-creates an unclaimed placeholder owner (invariant 6), so you need a handle-returning overload and `AssignFarmhand` auto-deletes the placeholder.

**The Steam64 bind needs no `STEAM_ACCOUNTS` entry.** Vanilla `authCheck` (`GameServer.cs:495-506`) admits a returning player to a farmhand purely on `farmhand.userID.Value == userID` (the connecting client's own `getUserID()`) — there is no account allowlist anywhere in mod or engine; `STEAM_ACCOUNTS` is the server's *own* bot-login credentials. Enforced only on Steam/GOG (`getUserID()` non-empty); LAN returns `""` → `authCheck` returns true for anyone, so the bind is identity-resume, not LAN access control (per [[abandoned-claim-is-steam-only]]).

**At finalize time the world is pre-reconciliation — don't resolve the pet with `Farmer.getPet()`; scan all locations.** The finalizer runs at `SaveLoaded`, *before* the reconciliation chain and *before* the day-start `dayUpdate` that warps the pet to its Farm bowl. `Farmer.getPet()` scans only `Game1.getFarm().characters` then each farmer's *resolved home* — so a pet that deserialized into an interior (a FarmHouse/Cabin) at that instant is in neither list and `getPet()` returns **null intermittently**. Resolve it with a full-world scan instead (`Utility.ForEachLocation(includeInteriors: true)`, first `Pet` found) — the pet always exists somewhere in `Game1.locations`, and it is farm-scoped (one per save; `Pet` has no per-farmer owner field, only a bowl assignment), so the farm's pet is the demoted owner's. **Why:** the pet-relocation E2E flaked ~38% with the finalizer running but emitting `npcsMoved:0`; root cause was `getPet()`'s location-order-dependent scan at the pre-reconciliation finalize instant. The same caution applies to any household NPC: prefer an all-locations scan over a helper that assumes the post-reconciliation/post-dayUpdate world.

**Cellars are per-farmer-slot, not master-only — a host swap must MOVE the contents, not warn-and-accept.** SDV pre-creates one cellar per farmer slot at load (`"Cellar"`-1 + `"Cellar2".."CellarN"` up to `HighestPlayerLimit`, `Game1.cs:7417-7422`) and assigns them by UID (`updateCellarAssignments`, `Game1.cs:4515`); `Cabin` inherits `GetCellarName()`, so **cabins DO get cellars** — "cabins have no cellar / there is one master cellar" is wrong. `"Cellar"`-1 is hardwired to the master, and the demoted owner's casks physically live there from when they were master; cellar contents are **location-bound**, so a swap reassigns the owner to a *different, empty* `CellarN` and the casks stay in the (now Server-bot) `"Cellar"`-1 unless moved. Resolve the destination via `updateCellarAssignments()` (idempotent — the engine calls it at load/day-start/join) then `Cabin.GetCellarName()`, and move the objects across. The assignment is **stable** across later day-start reshuffles because `updateCellarAssignments` skips slots whose holder still resolves via `GetPlayer` — which resolves *offline* farmhands (`Game1.cs:10937`, `onlyOnline=false` default), so the customized owner keeps their slot. **Why:** I documented the cellar as a "known limitation" (out of scope to move); the user rejected it ("makes no sense to treat it differently than the farmhouse interior — clearly a gap"), and decompiled source proved the master-only belief wrong (adjacent to [[masterplayer-is-player-on-server]] but distinct: the trap here is *per-slot cellars exist*, not host-vs-master identity).
