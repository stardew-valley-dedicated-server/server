# Prefer the simplest, most direct solution

If the fix is to remove a line, remove it. Don't add a redirect, wrapper, or abstraction layer. If the codebase already has a pattern for something, follow it exactly.

**Why:** Three failure modes around scaffolding and abstraction:
- **Over-engineering when a one-liner works.** Drift toward "flexible" wrappers, redirects, or new abstraction layers when the codebase already has a working pattern (or when "delete this line" is the actual fix).
- **Building infrastructure with no consumer.** Caught 2026-04-24 on a planned `DiscoverTestIdentities` API + theory display-name porter to seed per-test entries in `ctrf-report.json` — a subagent check confirmed Makefile, LLMRenderer, UI, GitHub Actions, and FlakinessTracker all read only `results.summary` (never `results.tests[]`). Cutting the unbacked scaffolding saved ~320 LOC.
- **Dissolving an established home for a convention.** Caught 2026-05-05 on `Helpers/AssertHelpers.cs`. I recommended inlining the last remaining method into its single caller. That would have dissolved the established home for custom asserts in a codebase that demonstrably accumulates them. The "I'll re-extract when the next consumer arrives" framing is a lie — future-you misses the prior convention, the next assertion lands at a new call-site, and the pattern fragments until a multi-day consolidation pass is needed.
- **Reflex-rejecting a user-proposed consolidation as "scope creep."** When fixing a flaky `caveChoice` handler I had to move it off a racy loop; the user asked whether the *sibling* handler (`HandlePetChoice`, same accidental placement, same drift) should move too. I twice waved it off as "scope creep" and argued from a stale adjacent comment ("deliberately deferred / menu-aware") instead of checking. The user corrected both: merging the two into one shared `PreSeedChoices()` home is the *consistency / anti-drift* fix, not creep — and the safety question must be **verified**, not assumed. Reading the relocated method (its reflection path touched no menu) + `git log` (the loop placement was accidental, not designed) settled it in minutes. "Prevent scope creep" must not become "skip necessary consolidation that keeps the codebase clean."

**How to apply:**
- **Default to the simplest fix.** Before proposing a wrapper, redirect, or new abstraction, ask: does an existing function or pattern in this repo already do this? If yes, use it. If the simplest fix feels too small, that's usually correct.
- **Verify a downstream consumer before scaffolding.** When a refactor proposes new infrastructure to *feed* something (per-test entries in a report, extra fields on an event, a new index), grep for an actual reader before building it.
- **Pattern continuity beats consumer count.** The "verify a consumer" check applies to *new* infrastructure. It does NOT apply to an existing "home for X" file (`*Helpers`, `Asserts`, `Validators`, etc.) whose member count has shrunk to one. Signals it's a home: plural name, single-concern namespace, sibling files suggesting more will accrete. If yes, the single consumer is enough — keep the home, add to it.
- **A user-proposed consolidation is not scope creep — verify it, don't reflex-reject it.** When the user asks to fold sibling code together (move a second handler onto the same path you're already fixing, dedupe parallel logic into one home), that's consistency/maintainability work, not creep. Don't decline it by reflex or by quoting an adjacent comment — resolve the only real question, *does the moved code still work at the new call site?*, by **reading** the code it touches (and `git log` for whether the old placement was designed or accidental). Decline only if reading shows a real dependency the move would break, and say which. Scope creep is the *silent, unannounced* addition during a signed-off review cycle ([`plan-discipline.md`](plan-discipline.md)) — a consolidation the user explicitly requested is the opposite of that. Preventing creep must never become skipping necessary cleanup.
- **"Simplest" means simplest that meets the *stated* constraints.** When the user has declared a constraint — perf, GC, latency, security — the simplest solution is the simplest one that satisfies it, not the fewest lines ignoring it. On game-thread mod hot paths, where GC is a from-the-start design constraint here, the allocation-minimal version (buffer reuse, swap-buffers) IS the qualifying simplest — see [`mod-game-thread-allocation.md`](../mod-game-thread-allocation.md). This is not a licence to optimize speculatively: it applies only where the constraint is actually declared and the code is on the hot path.
