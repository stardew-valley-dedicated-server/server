# Verify a new fail-fast preflight check against the repo's committed config, not just the documented example

When you add a `throw` (or other fail-fast rejection) at startup, in CI, or in a build script that rejects an input shape — env var values, file presence, version assertion, capacity ratio, manifest field — open the actual checked-in config and confirm it still passes. The example/template (`*.example`, default fixtures, sample `appsettings.json`) documents the *intended* shape; the committed config is what the next run actually sees. They drift.

**Why:** Implemented a plan's option-(b) tightening verbatim — a preflight throw on a config ratio, with the plan-author's note "may over-reject some valid configs, revisit if operators hit false positives." I treated that as a future-tense risk and shipped the throw. The repo's committed config violated it; the very first run after merge would have aborted CI. User caught it before merge ("we only have 2 of those — that's intentional"). The fix was downgrading the throw to a diagnostic event, since the surrounding system was structurally fine under the rejected config — what looked like a bug to reject was correct behavior. The same shape applies to any preflight: mod startup checks (SMAPI `ModEntry` config validation), tooling preflights (`tools/docker-modern/` build asserts), CI-gate scripts, and test-infra capacity checks all share the failure mode — the rejection logic gets written against the example, not the reality.

**How to apply:** Before landing any new fail-fast rejection, open every committed config it gates and confirm they all pass. If any violate the new check, the throw is wrong — either weaken it to a warning/diagnostic emit, fix the committed config in the same change, or scope the check more precisely to the case it actually needs to catch. Yellow-flag triggers in plan or PR text: "may over-reject", "conservative", "revisit if false positives", "first cut" — these are not future-tense risks, they are present-tense bugs to verify against today's config before merge. Distinct from `verify-claims.md` (verify named claims) and `verify-documented-config-is-consumed.md` (config knobs have consumers): this rule is about new *rejections* needing to be tested against existing valid inputs.
