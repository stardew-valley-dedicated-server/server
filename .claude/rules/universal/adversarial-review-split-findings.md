# Adversarial review must split findings, and must not collapse valid recommendations along with their weak framing

Adversarial self-review can fail in two specific ways. **(1) OOS-as-cover:** every finding gets labeled either inherent non-issue (provably safe — e.g. no-op base call today, .NET memory-model guarantee on x64) or OOS deferral, and OOS gets used to ship known small bugs that are right there. **(2) Recommendation collapse:** when adversarial pushback finds a real weakness in your *framing*, you withdraw the underlying *claim* too, when often only the framing was weak.

**Why:**
- **OOS-as-cover.** A post-implementation review listed seven "considered non-issues"; the user asked which were inherent vs out-of-scope, and an honest split revealed three were real bugs I'd minimized as "OOS" — all small and adjacent (one `volatile`, one missing `base.DisposeAsync()`, ~10 lines of eager run-id seeding), each with a plausible deferral excuse ("pre-existing", "not asked for"). All three would have shipped without the split prompt.
- **Recommendation collapse.** In a rules-review session I recommended folding `verify-documented-config-is-consumed` into `verify-claims`, then under adversarial review found the token-saving argument thin and withdrew the fold *entirely* — same shape with two L1→L2 rescopes. The user had to push back ("if that's cleaner, let's do it") to recover the valid underlying claim. The structural failure: adversarial review correctly weakens *one* of {framing, claim}, but I let it weaken both.

**How to apply:**
- **Split findings.** During self-review, label each as inherent non-issue or OOS deferral. For inherent non-issues, name *why* it's safe (the guarantee, the empty stub, the platform constraint) — if you can't, it's not inherent. For OOS deferrals, name the cost of fixing now: lines of code, files touched, whether investigation is needed. If the answer is "small and adjacent," fix it in this task. Don't use "the plan didn't ask for it" as cover.
- **Separate framing from claim.** When adversarial review weakens your own recommendation, ask explicitly: "is the *underlying finding* weakened, or only the *framing/justification*?" If only the framing — restate the claim with sharper framing and keep it. If the claim itself — drop it. Don't withdraw both as a package; that's the same OOS-as-cover failure mode in a different costume ("on second thought, not worth it" instead of "out of scope").
- **In a fan-out → verify → synthesize workflow, a binary verdict filter reproduces recommendation-collapse at scale.** When synthesis consumes only the `CONFIRMED` survivors, a `REFUTED` parent finding silently takes its *verified sub-findings* down with it (the overstated parent claim refuted correctly, but a small true dead-code claim in its evidence was real). Two safeguards: (1) have the verifier emit `salvagedSubFinding` / `actualLinesSavable` even on `REFUTED` verdicts and feed those to synthesis — never drop a refuted finding's body wholesale; (2) don't rank candidates purely by `lines × confidence` — reserve slots for small-but-HIGH-confidence dead-code wins, which it otherwise crowds out. And per [`subagent-findings-are-claims.md`](subagent-findings-are-claims.md), re-verify the *raw* finding set after synthesis — it only ever saw what the filter passed.
