# Adversarial review must split findings, and must not collapse valid recommendations along with their weak framing

Adversarial self-review can fail in two specific ways. **(1) OOS-as-cover:** every finding gets labeled either inherent non-issue (provably safe — e.g. no-op base call today, .NET memory-model guarantee on x64) or OOS deferral, and OOS gets used to ship known small bugs that are right there. **(2) Recommendation collapse:** when adversarial pushback finds a real weakness in your *framing*, you withdraw the underlying *claim* too, when often only the framing was weak.

**Why:**
- **OOS-as-cover.** Finishing an abort-path summary-writer task: my post-implementation review listed seven "considered non-issues." User asked which were inherent vs out-of-scope. Honest split revealed three were real bugs I'd minimized as "OOS" — all small and adjacent (one `volatile`, one missing `base.DisposeAsync()` call, ~10 lines for eager run-id seeding). Each had a plausible-sounding deferral excuse ("pre-existing", "not asked for", "plan didn't anticipate it"). All three would have shipped without the user's split prompt.
- **Recommendation collapse.** Twice in a rules-review session: I recommended folding `verify-documented-config-is-consumed` into `verify-claims`, then under adversarial review found the token-saving argument thin and recommended skipping the fold entirely. Same shape with two L1→L2 rescope recommendations: the cost-saving framing was weak under adversarial pressure, so I withdrew the rescope itself. User had to push back ("if that's cleaner, let's do it") to recover the underlying valid claim. The structural failure: adversarial review correctly weakens *one* of {framing, underlying claim}, but I let it weaken both.

**How to apply:**
- **Split findings.** During self-review, label each as inherent non-issue or OOS deferral. For inherent non-issues, name *why* it's safe (the guarantee, the empty stub, the platform constraint) — if you can't, it's not inherent. For OOS deferrals, name the cost of fixing now: lines of code, files touched, whether investigation is needed. If the answer is "small and adjacent," fix it in this task. Don't use "the plan didn't ask for it" as cover.
- **Separate framing from claim.** When adversarial review weakens your own recommendation, ask explicitly: "is the *underlying finding* weakened, or only the *framing/justification*?" If only the framing — restate the claim with sharper framing and keep it. If the claim itself — drop it. Don't withdraw both as a package; that's the same OOS-as-cover failure mode in a different costume ("on second thought, not worth it" instead of "out of scope").
