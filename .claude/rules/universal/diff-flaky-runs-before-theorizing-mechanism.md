# For a flaky failure, diff one signal across passing and failing runs before theorizing a mechanism

When a test (or any process) fails intermittently and you have artifacts from both passing and failing runs, the runs ARE a built experiment: pick one observable signal and tabulate its value across several passes and the fail(s). A signal that splits cleanly along the pass/fail line is the differentiator — it points at the cause directly. Do this *before* constructing a mechanism story; a plausible mechanism is cheap to invent and expensive to chase down the wrong layer.

**Why:** Diagnosing the `AbandonedClaim_SweptOnReload` flake (33%), I burned a long session on four plausible-but-wrong mechanism theories in sequence — disk-flush lag, reload-interrupts-save, `_old`-backup fallback, save-rename interleaving — each refuted only after detailed log-staring. What actually cracked it in one step: tabulating a single signal ("what in-game date did the reload load?") across 4 passing + 1 failing run. The result was a perfect 5/5 split (pass→day 2, fail→day 1), which immediately localized the cause to `/newgame`'s gating `SaveLoaded` firing at day 0 vs day 1 — no mechanism guessing needed. Every minute spent theorizing before running that comparison was wasted.

**How to apply:** The moment you have ≥1 passing and ≥1 failing artifact for the same flaky case, list candidate signals (a timestamp ordering, a logged date/count/id, which code path fired, an event's presence/absence) and grep each across all available runs into a small pass-vs-fail table. A clean split is the lead — chase that, not your first mechanism hunch. If no single signal splits, widen the signal set rather than falling back to speculation. Distinct from [`isolate-pipeline-stage-before-fixing.md`](isolate-pipeline-stage-before-fixing.md) (bisecting stages *within one run* of a producer→consumer pipeline) and [`runtime-post-conditions-are-gates.md`](runtime-post-conditions-are-gates.md) (verify a root-cause hypothesis by *running* it): this rule is about *forming* the hypothesis from the pass/fail population you already have.
