# A probe that detects another component's state must mirror that component's full resolution logic

When you write detection code that models another component's behavior — "is this container configured the way the real X would interpret it?", "what would the runtime resolver decide?" — the model must mirror the target's complete resolution order, not just the path you happen to be familiar with. Picking only the happy path produces silent false negatives: the probe says "no conflict" while the target component is actively in the state you were trying to detect.

**Why:** The Steam-account conflict probe in `tests/JunimoServer.Tests/Containers/SharedSteamAuth.cs` was supposed to fail-fast when a non-test sidecar on the same Docker host was already configured with a Steam username the test planned to use. First version of `ExtractUsernamesFromEnv` read only `STEAM_ACCOUNTS` from inspected production-container env. But the real sidecar (`tools/steam-service/Program.cs:62-114` `DiscoverAccounts`) tries `STEAM_ACCOUNTS` JSON *first*, then falls back to legacy `STEAM_USERNAME` for account 0. Production deployments use the legacy fallback path by default (per `docker-compose.yml`); the probe's partial model never saw their username. Tests passed the probe, then collided at runtime with `LogonSessionReplaced`. Fix: rewrite `ExtractUsernamesFromEnv` to walk the env once, capture both keys, and apply JSON-first/legacy-fallback resolution that mirrors the real sidecar step-for-step.

**How to apply:** When the detection code's job description is "predict what component X would do with this configuration," open X's resolver and read it end-to-end before writing the probe. Specifically: list every input X consults, every fallback chain, every "if A is set use A, else use B" branch — your probe must implement all of them or it will under-detect. Distinct from `verify-claims.md` (verify named claims you publish) and `subagent-findings-are-claims.md` (verify claims you receive): this rule is about *modelling logic from another component*, where the model has to be a faithful subset of the target's resolution path, not a convenient one. Put a comment in the probe naming the source-of-truth file and line range so the next reader knows to re-sync if the target's resolution order changes.
