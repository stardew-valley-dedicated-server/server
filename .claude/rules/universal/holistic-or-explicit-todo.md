# Don't hedge with empty scaffolding — either build the holistic solution or mark a clear TODO

When a task calls for a "clean and consistent" or holistic solution, don't ship half-built scaffolding to *imply* the missing piece will arrive later. If something must genuinely be skipped, leave a TODO that names the gap concretely — not a comment that softens the omission ("future work", "consumers may want this", `field = []` placeholders). Either complete the piece, or be explicit that it's missing and why.

**Why:** Caught twice in one session. First when I proposed merging `ctrf-report.json` with `results.tests = []` plus a "future work" comment — the empty array would silently misrepresent the merged report as containing no tests. Second when the user pushed back generally: "Don't skip anything if it's needed for a clean and holistic solution. IF we skip something, leave a clear TODO." The failure mode is producing artifacts that *look* complete but carry hidden gaps a downstream consumer won't notice until they hit the missing data.

**How to apply:** Before shipping an empty placeholder field, an empty array stub, or a comment like "future" / "for now" / "consumers may want", ask: does this look complete to the next reader? If yes and it isn't, either build it or omit the field entirely so the gap is visible. When you must skip, write `// TODO:` with the concrete missing thing — what's not implemented, what would be needed to implement it, what the current behavior gives instead. Distinct from `simplest-solution.md`'s "verify a consumer before adding scaffolding" (that rule says cut unbacked features); this one covers what to do when the feature is genuinely needed but you can't finish it now.
