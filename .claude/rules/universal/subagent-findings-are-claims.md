# Subagent findings are claims to verify, not ground truth

When a subagent (Explore, Plan, general-purpose, etc.) returns a confident bottom-line conclusion that a downstream decision will rest on, treat that conclusion as a *claim* with the same verification bar as a claim you'd write yourself — especially identifiers, file paths, "X inherits Y", "X uses Y file", "no consumer of Z". Open the cited file:line and confirm the reading before building a plan on top of it. Distinct from `verify-claims.md` (which covers claims *you* publish): this covers claims *you receive*.

**Why:** An Explore agent reported confidently that the test-client image inherited `docker/rootfs/etc/services.d/xvnc/run` from the base rootfs. I built and shipped a whole plan on that bottom-line; only an end-to-end smoke test (`docker cp` from the produced image) surfaced the truth — `Dockerfile.test-client` doesn't `COPY docker/rootfs/` at all, and the file is a symlink owned by the base image. A one-second `Grep` of `Dockerfile.test-client` for `rootfs` would have caught it. Cost: a wrong-file edit, a useless rebuild, and a re-plan after the user had already approved.

**How to apply:** When a subagent returns a finding that a plan, a recommendation, or an edit will rest on, identify the *load-bearing* claims (the ones that, if false, invalidate the plan) and verify each one directly — open the file, grep the identifier, read the COPY/IMPORT/USING line. The agent's confidence is not evidence; its citations are. If the agent didn't cite a file:line for a claim that drives the decision, that's the gap to close. Cheaper to spend 30 seconds on `Grep` / `Read` than to discover the false premise after the artifact ships. Especially watch for "X inherits Y", "X uses Y file", "Y is provided by Z" — relational claims about how systems compose are the most failure-prone shape because they require reading multiple files to confirm, and the agent often only read one.

**Negative-existence claims from scope-limited agents are blind beyond their slice.** An agent assigned one area of a corpus cannot see the rest, so its "X is undocumented", "no consumer", "doesn't exist" conclusions are structurally unreliable: a docs-area reviewer assigned `docs/admins/` reported STEAM_KEEP_LANGUAGES as "undocumented" when it had a full section in `docs/developers/architecture/steam-auth.md` — and the correct fix changed once the full tree was checked (a cross-link, not new duplicate prose). Before acting on any negative-existence claim from a sliced fan-out, grep the *whole* corpus yourself, and expect the remediation to change shape when the claim only half-holds.
