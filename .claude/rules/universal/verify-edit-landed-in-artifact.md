# Verify your edit landed in the produced artifact, not just that the build succeeded

A green build does not prove your source change is in the binary. When a build pipeline `COPY`s files conditionally — a multi-stage Dockerfile that only copies a subset of the rootfs, a multi-project solution that only includes some `.csproj` outputs — your edit can be to a file that the targeted artifact never consumes. The build succeeds because the consuming stages didn't change; the artifact is identical to before; you proceed thinking the change is in. It isn't. Inspecting the produced artifact directly (e.g. `docker create` + `docker cp`, `dotnet nuget locals` cache check, `unzip -p` on a packaged binary) is the cheap check that catches this before any runtime verification.

**Why:** Edited `docker/rootfs/etc/services.d/xvnc/run` and ran `make build-test-client`. BuildKit reported success — every layer cached, image tagged, all green. I assumed the edit was baked in. It wasn't: `Dockerfile.test-client` doesn't `COPY docker/rootfs/` (only `Dockerfile` does, for the server image). My edit was to a file the test-client image never sees. The test-client's actual `/etc/services.d/xvnc/run` is a symlink owned by the base image, untouched by my edit. A `docker cp <image>:/etc/services.d/xvnc/run -` after the build would have shown a symlink with no relation to my source change in 5 seconds. Without that step, I lost a full plan-and-implement cycle to a confident-but-wrong assumption.

**How to apply:** After a build that's supposed to incorporate a specific source edit, spot-check the artifact for the change before treating "build green" as success. For Docker: `docker create <tag>` + `docker cp <id>:<path> -` (pipes the file out as a tar; `MSYS_NO_PATHCONV=1` on Windows Git Bash to defeat path mangling) — read the bytes you expected to land. For .NET / packaged outputs: `unzip -p`, `dotnet build --verbosity normal` to see actual file paths copied, or open the produced DLL/jar with the appropriate inspector. Especially important when (a) build cache is involved, (b) the project has multiple Dockerfiles or assemblies that selectively pick from the same source tree, or (c) the change is to a file whose consumer relationship you haven't personally verified. Distinct from `runtime-post-conditions-are-gates.md` (which covers verifying *runtime behavior* against post-conditions): this rule is one step earlier — proving the change is even in the binary that runtime would run.
