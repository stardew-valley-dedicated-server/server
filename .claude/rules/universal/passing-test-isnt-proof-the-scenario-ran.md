# A green test proves its assertions held — not that the scenario you described actually happened

When a test passes, that confirms the assertions evaluated true. It does NOT confirm the *scenario under test* occurred — an assertion can be satisfied by a path that bypasses the very thing you meant to exercise. Before reporting a test as validating a behavior, read the run's own artifact (server/container log, recording, event stream) and confirm the intended sequence of events actually took place. "passed: 1" is necessary, not sufficient.

**Why:** A two-client "two same-day weddings" E2E passed three consecutive runs while only **one** ceremony ever fired. The assertion gated on `FarmhandIsMarried` for both farmhands — but the marriage `Status=Married` is set at the day-transition finalizer *before any ceremony plays* (`Game1.cs:7961`), so both read married even though the second wedding's ceremony never ran. I reported "both ceremonies completed — no hang" off the green checkmark; the user forced me to read `containers/server-0/container.log`, where only one "Readied wedding wait gate" line and timestamped joins **75s apart** showed the scenario I claimed never happened. The assertion was technically true and totally uninformative about the behavior under test. (Related: the gate was later switched to a durable, ceremony-only signal — both spouses warped to `Farm`, which only `endBehaviors` does — precisely so passing *means* the ceremonies ran.)

**How to apply:** After a test goes green for a behavior you're about to report as fixed/covered, ask: "could this assertion be satisfied *without* the scenario running?" If yes — and especially for multi-actor, multi-phase, or timing-dependent scenarios — open the run artifact and confirm the expected events fired, in the expected order, the expected number of times (grep the log for the per-event marker; check timestamps for "concurrent"/"sequential" claims). Prefer assertion signals that are *produced only by the path under test* (a ceremony-only side effect) over signals a bypass path can also set (a status flag flipped elsewhere). Distinct from `runtime-post-conditions-are-gates.md` (which says *run* the thing rather than statically reviewing) and `verify-edit-landed-in-artifact.md` (which says confirm your edit is in the *binary*): this rule says even after running, confirm the *run exercised the scenario* — read the artifact, don't trust the pass.
