# Size workflow verification to batched verifiers, not per-finding fan-out

In multi-agent workflows, default verification to ONE batch verifier per producer (verifying all of that producer's findings in a single agent with indexed verdicts), not one verifier agent per finding. Reserve per-item fan-out for cases the user explicitly asks to scale ("verify each independently", a stated token budget that warrants it).

**Why:** The docs-audit workflow followed the Workflow tool's canonical review→verify pattern literally — one adversarial verifier per finding — and was heading past 100 agents before the user stopped it mid-run ("we dont need SO many subagents, drop some"). Rebuilt with one batch verifier per review area (~13 verifiers, ~27 agents total), verification quality held: every verdict still re-derived evidence, adjusted priorities, and salvaged sub-findings; zero refuted-in-error. Per-finding fan-out multiplied agent count ~8× for no measured gain. Ultracode/effort settings raise thoroughness, not the agent multiplier.

**How to apply:** When writing a workflow script with a verify stage, make the unit of verification the producer's output batch (`agent(batchVerifyPrompt(findings))` with index-keyed verdicts), and tell the verifier to spend depth proportional to severity (deep on accuracy/Critical-High, premise-check on editorial). Estimate worst-case agent count (producers × findings-per-producer) before launching; if it lands in the many-dozens, restructure to batches first rather than waiting to be stopped. Resume via `resumeFromRunId` after downsizing — completed producers return from cache.
