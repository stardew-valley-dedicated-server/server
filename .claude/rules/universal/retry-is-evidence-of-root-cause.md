# A retry loop is evidence of an unresolved root cause, not load-bearing infrastructure

When you encounter a retry loop, exponential backoff, "transient error" classifier, or a `try/catch` that swallows-and-continues, treat it as a *symptom* of a root cause that hasn't been fixed yet — not as the correct shape of the code. Before extending it (adding a new error type, raising the attempt count, applying it to a sibling call site), ask: what is the underlying failure being papered over, and can it be fixed at source? Only extend the band-aid if (a) the root cause is provably outside our control (third-party API, real network flake) and (b) the cost of fixing it exceeds the cost of the retry. Otherwise, fix the root cause and delete the band-aid.

**Why:** E2E tests intermittently failed at container creation. `ClientPool.CreateClientGuardedAsync` already had a 3× retry loop with an `IsTransientDockerError` filter, and my first instinct was to add the same retry server-side. The user pushed back twice ("hint: retries", "no bandaids… a clean and consistent solution") before I found the root cause: Testcontainers 4.3.0's default 100ms `namedPipeConnectTimeout`, which fails under parallel-startup pressure on Windows. The retry's own comment named the symptom ("Docker Desktop's named pipe can be slow") but not the knob — treating it as load-bearing kept me from investigating until forced. The fix was 6 lines of `.WithDockerEndpoint(...)` and *deleting* the retry loop.

**How to apply:** When you see a retry / fallback / classifier in code you're about to extend or copy, stop and read the surrounding comments and commit message for *why* it exists. If the reason names a symptom but not a root cause ("under load", "sometimes flaky", "transient"), that's the tell — investigate the root cause before adding any code. A useful prompt for yourself: "if I fixed the underlying problem, would this retry become dead code?" If yes, fix the underlying problem. The retry's existence is also a hint that the root cause was *known* to a past contributor but not pursued — the cheap fix often won. Don't repeat that trade-off without revisiting it.
