# Runtime evidence outranks plan claims — post-conditions, fix predictions, and root-cause hypotheses are all gates

When a plan's post-conditions read as runtime observations — "X returns…", "Y works end-to-end against…", "Z appears in the output" — those are *gates*, not aspirational text. Static review (`dotnet build`, `grep` that references are gone, reading the diff) does not satisfy them. Run the thing and observe the result before reporting complete. This outranks the plan's own claims: a received plan's prediction ("Change 1 drops the failure rate") is a claim to verify against runtime data, and if the data contradicts it the task isn't finished — investigate further, even outside the original scope.

**Why:** Plan `02-jsonl-reporter-deletion`'s post-conditions named a runtime check ("`make test-events` returns phase-decorated rows"). I ran a static sweep (build clean + zero refs + diff coherence) and asked "done?"; the user pushed back twice, and the runtime walk surfaced two real bugs in a jq filter I'd transcribed verbatim from the plan — it crashed on null `displayName` and never matched (actual names are fully-qualified `Namespace.Class.Method`). The same failure recurred two more ways, each caught only by user pushback: a recording-pipeline plan whose own fix-prediction was the wrong premise (runtime ffprobe showed the `finalizedSegments=0` rate had *risen*, not fallen — root cause was upstream in `HostStartTicks`), and a *diagnosis* plan where a 2-minute experiment (`CLIENT_FPS=10`) would have falsified the load-bearing root-cause hypothesis before ExitPlanMode. Post-conditions, a fix's predicted effect, and a diagnosis's root-cause claim are all runtime gates. Distinct from `plan-discipline.md` (verifying a plan's claims *before* ExitPlanMode); this is verifying *outcomes*.

**How to apply:** At "all done?", re-read the post-conditions verbatim. For each, classify: static fact (compiles, refs gone, file exists) → build/grep/read confirms it; runtime observation (returns X, works against Y, appears in Z) → actually run it on real data and inspect. A user-facing recipe (Make target, CLI, query) doubles as its own smoke test. If a runtime check genuinely costs more than ~2 min (full E2E, multi-host) and you can't run it, say so explicitly — "post-condition X not exercised, needs <cost>" — rather than implying via silence that all were checked.
