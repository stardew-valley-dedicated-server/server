# Runtime evidence outranks plan claims — post-conditions, fix predictions, and root-cause hypotheses are all gates

When a plan's post-conditions section is phrased as a runtime observation — "X returns…", "Y works end-to-end against…", "Z appears in the output", "the runbook steps run cleanly" — those are *gates*, not aspirational text. Static review (`dotnet build`, `grep` confirming references are gone, reading the diff) is not enough to satisfy them. Run the thing and observe the result before reporting the plan complete.

**Why:** Finishing plan `02-jsonl-reporter-deletion`: post-conditions explicitly said "`make test-events TEST=Class.Method` returns typed events… phase-decorated rows" and "runbook steps work end-to-end against a deliberately-failed test." I read those as guidance, ran a static sweep (build clean + zero `_jsonlReporter` refs + diff coherence), and asked "ready to mark this complete?" The user pushed back twice — first asking me to verify post-conditions, then explicitly "yes, run them" — and the runtime walk surfaced two real bugs in the jq filter I had transcribed verbatim from the plan: it crashed on null `test.displayName` and never matched because actual displayNames are fully-qualified (`Namespace.Class.Method`), not `Class.Method`. Both would have made the failure-debug runbook useless on first real failure post-merge. Distinct from `plan-discipline.md` (which covers verifying a plan's *claims* before ExitPlanMode): this is about verifying a plan's *post-conditions* before declaring it complete.

**The plan can be wrong about what its own fix delivers — runtime evidence outranks plan claims.** Same shape, different incident: implementing a "fix for fixes" recording-pipeline plan, post-conditions §3-§5 named runtime ffprobe checks ("`nb_read_frames ≈ round(durationSec * fps)`", "distribution of `finalizedSegments` shifts away from 0"). I shipped 6 file edits, ran static checks (3 builds clean, grep clean), and reported "done" without exercising the runtime gates. User asked "review the latest test run" — only then did I run ffprobe and discover all four canary clips still produced 1 frame / 1.0s and the `finalizedSegments=0` rate had *risen* (81% vs the 69% baseline the plan was meant to fix). The plan's own predictions about Change 1's effect were the wrong premise — the actual root cause was upstream, in `HostStartTicks` anchoring, not in the slack widening or split-clamp the plan addressed. A received plan's post-conditions are a contract; if the runtime check fails, the conclusion is "this plan didn't deliver", not "the plan did its part, the rest is out of scope."

**Diagnosis plans need the same gate — validate the root-cause claim, not just the fix's post-conditions.** Same shape applied one layer earlier: investigating a per-test clip showing only title screen when the test had reached gameplay, I built a plan around the hypothesis "FpsThrottle gating BeginDraw → stale framebuffer." Static evidence (FpsThrottle code path, recorder fps coupling, ffprobe of frame sizes) was consistent with that hypothesis, but also consistent with two competitors (CPU-starved Update during loading, x11grab phase aliasing). I called ExitPlanMode citing the static evidence. User pushback: "No assumptions, validate everything. Run experiments and read the video files if you need to." A single 2-minute test run with `CLIENT_FPS=10` would falsify or confirm the load-bearing claim — and did, on the next attempt: the new clip showed gameplay frames. Without that step the plan could have shipped a fix targeting the wrong hypothesis (e.g. lifting throttle when the real issue was CPU starvation), forcing a re-investigation after merge. Diagnosis-plan claims are predictions about *why* the bug happens; the same "runtime evidence outranks plan claims" rule applies — if a cheap experiment can distinguish hypotheses, run it before ExitPlanMode.

**How to apply:** When you reach "all done?" on a plan, re-read its post-conditions section verbatim. For each bullet, ask: "is this a static fact (compiles, no references remain, file exists) or a runtime observation (returns X, works against Y, appears in Z)?" Static facts can be confirmed by build/grep/read. Runtime observations require actually running the command on actual data and inspecting the result. If the plan ships a user-facing recipe (Make target, CLI command, query), the runtime walk doubles as a smoke test for the recipe itself. If running it costs more than ~2 minutes (full E2E suite, multi-host scenario) and you genuinely can't, say so explicitly — "post-condition X (runtime check) not exercised, would need <specific cost>" — rather than implying via silence that everything was checked. Applies equally to plans you authored and plans you received: a received plan's prediction ("Change 1 should drop the failure rate substantially") is a claim to verify against runtime data, not a result to assume. If the runtime data contradicts the plan's prediction, the task is not finished — investigate further, even if it pulls scope outside the original plan.
