# Prefer the live event stream over disk artifacts when both carry the same data

When you need to surface runner data in the UI, check whether a live WebSocket / IPC event already carries it before reaching for the on-disk artifact. The runner often writes the same data to both — `infrastructure.jsonl` on disk *and* a typed `instance_*` event over the WebSocket. Picking the disk artifact when a live event exists creates a second source of truth: the UI now polls and parses what the store already has reactively.

**Why:** Building the test-UI infrastructure timeline, I read `EventNames.cs` and `infrastructure.jsonl`'s schema and concluded "events flow to disk, fetch from disk." I never asked whether those events were also on the WebSocket the UI was already listening to. They were — the store has handlers for `instance_created/leased/returned/disposed/poisoned/...` that populate `state.instances[].history` in real time. The wrong choice forced a JSONL fetcher composable, polling, `If-Modified-Since` plumbing, and (worst) a translation map between two parallel vocabularies — `server_*`/`client_*` on disk vs `instance_*` on the WebSocket — because the broker emits different event names to each sink. The "tell" was finding myself writing `EVENT_TO_CATEGORY` to bridge them: a translation layer that exists only because I picked the wrong source.

**How to apply:** Before fetching a runner artifact from `/artifacts/` for a UI surface, grep `tests/test-ui/src/composables/useTestStore.ts` for handlers that already consume the same data type (`case 'instance_*'`, `case 'test_*'`, etc.). If a handler exists and populates state the UI can read, use that — no fetch, no polling, no parser. Only reach for the disk artifact when the data is genuinely not on the wire (e.g. `summary.json`'s `abortReason`, `container.log`'s contents). Yellow flag: if you catch yourself writing a map between two event-name vocabularies for "the same lifecycle," stop — you're consuming the wrong source. Distinct from `runner-ui-pipeline-plumbing.md` (which covers adding *new* fields through the IPC chain): this rule is about not re-fetching data that's already arriving.
