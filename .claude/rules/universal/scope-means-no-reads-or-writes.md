# "Don't touch X" means no reads or writes, including link repair inside X

When the user excludes a directory or file from a task's scope, treat it as fully read-only for the duration of the task. Do not propose edits inside it, even to repair references your own changes would otherwise break. Resolve such conflicts on your side (preserve the link target) rather than theirs (rewrite their link).

**Why:** Mid-plan I proposed updating a citation inside `.claude/tasks/working/12-join-phase-flake-diagnostics.md` to point at a renamed rule. User clarified that `.claude/tasks/` was out of scope and the plan should not touch it — meaning even a surgical one-line link-repair edit was off-limits, not just behavioral changes. That reframed the whole plan: instead of "rename the rule and update the citation," the answer was "preserve the original filename so the citation still resolves." Misinterpreting "don't touch" as "don't change behavior" loses the constraint's actual force.

**How to apply:** When the user scopes part of the tree out, that tree becomes read-only. If a refactor in scope would invalidate a reference inside the out-of-scope tree, the burden is on the in-scope change to preserve the contract (keep the filename, keep the export, keep the message format) rather than on the out-of-scope tree to adapt. Surface the trade-off in the plan ("X kept its old name to preserve citation in Y, even though the new content suggests a different name") rather than papering over it.

**Concrete corollary — preserve cited filenames during refactors.** Before any rename/delete in a refactor, grep the project for the filename. For every hit, ask: is the file containing this reference inside the scope I'm allowed to edit? If no, the original filename must survive — rewrite contents in place rather than rename. The same `.claude/tasks/working/12-join-phase-flake-diagnostics.md:166` citation that established this rule would have been silently broken by a `debugging.md` → `logging-in-mod.md` rename in a prior cleanup pass. The plan should explicitly document when a filename is kept for citation-preservation rather than fit (so the next reader knows the name is load-bearing, not chosen).
