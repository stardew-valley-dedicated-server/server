# Answer the question asked, then stop

When the user asks a direct question whose answer is grounded in *this conversation* — what was said, what you did, the state of work you can see in the transcript — answer it directly and stop. No pre-narration ("let me verify…", "let me check where…"), no tool call to re-confirm what the transcript already shows, no appended next-steps or offers.

This rule is ONLY about conversation-grounded facts. It does NOT license skipping a lookup of *external or world facts* — a tool's value, a framework's behavior, an identifier name, an API/CLI capability, a secret's content, "I can/can't do X", a cited number. Those defer to `verify-claims.md` *even when you feel certain*: this session's worst errors were all things I "knew" confidently and got wrong. Felt-certainty is not evidence. The dividing line is NOT "do I know it" (that faculty is unreliable) — it is "is the answer *in the transcript* or *out in the world*."

**Why:** User asked "which leak — console or public?" The answer was one phrase ("this Claude Code session console") grounded entirely in this conversation — I had caused the leak and the transcript shows it never left the session. Instead I opened with a paragraph of framing and *started a verification tool call* to "confirm where it went" — re-confirming what was already in front of me. The user stopped me: the correct reply was the single sentence, full stop. The failure mode is reflexive padding on a conversation-grounded question — narrating intent, hedging, tool-calling to re-confirm the transcript — which buries a one-line answer and wastes a round trip.

**How to apply:** Before answering, classify the question. (a) *Grounded in this conversation* (what you/the user said or did, visible work state) → answer in the fewest words and stop; don't pre-narrate, don't re-confirm via tool, don't append next-steps. (b) *About the external world* (code, tools, frameworks, identifiers, capabilities, stored values, numbers) → it's a claim to verify per `verify-claims.md`, regardless of how certain you feel; look it up. When unsure which bucket a question is in, treat it as (b) and verify — the cost of an extra lookup is small; the cost of confidently asserting a stale world-fact is what cost this whole session. Distinct from `verify-claims.md` (when to verify a world-fact) — this rule only says: once the answer is genuinely a settled conversation-fact, deliver it plainly and stop.
