# Split a field only when a later write erases a still-needed earlier write

When two independent concerns share a single field AND writes from one concern overwrite writes from the other, lose information across reads, splitting into separate fields is the fix. The trigger is *information loss from concurrent writes*, not "this field has values from two categories."

**Why:** In the instance-status work, `status` encoded both health (`poisoned`, `idle`, `in_use`) and lifecycle (`disposed`). `instance_disposed` writes overwrote `status='poisoned'` with `'disposed'`, losing the poison bit across snapshots. A "poisoned wins" precedence hack in the label helper fixed display but left the state model lossy. The clean fix was splitting: `disposed: boolean` for lifecycle, `status` for health only. Status reports what the container *is*, `disposed` reports what happened *to* it, and the precedence hack was deleted.

**How to apply:** When you see writes to a single field that semantically encode different dimensions AND a later write overwrites an earlier one such that information is lost (a health enum being extended with lifecycle values that wipe the health, an ID field being overloaded with state that wipes the ID), propose the split. Priority-ordered enums and "X wins" precedence helpers are the symptoms — they exist to recover the lost dimension at read time. Splitting eliminates them.

**When this rule does NOT apply.** A single field whose values are points along a single trajectory (`pending → queued → running → finished`) is not "two concerns." Lifecycle-progression writes overwriting earlier lifecycle values is correct behavior; no information is lost because the earlier value is no longer current. Misinterpreting a transient state at read time (e.g. a sweep treating `"running"` as a signal to write `"canceled"`) is a different failure mode — fix the sweep's logic, not the field's shape. Per `rule-applies-only-when-failure-mode-matches.md`, match the incident, not the surface pattern.
