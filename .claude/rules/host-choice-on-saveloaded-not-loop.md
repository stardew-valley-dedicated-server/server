# A one-time host choice/state write belongs on the OnSaveLoaded pre-seed path, not the OneSecondUpdateTicked loop

A host setting that should be applied exactly once per save (a new-game choice, a one-shot world-state stamp) must run from the deterministic `AlwaysOn.OnSaveLoaded` pre-seed path (`PreSeedEarlyGameEvents`/`PreSeedChoices`), not from `OnOneSecondUpdateTicked`. On the once-per-second loop the write is not just slow (12s at `SERVER_TPS=5`, see [`one-second-update-ticked-fires-per-game-tick.md`](one-second-update-ticked-fires-per-game-tick.md)) — it **races any other writer of the same field** (a `/test` seed, an operator command, the engine) and silently overwrites whichever landed first. The pre-seed path runs synchronously inside `SaveLoaded`, which the `/reload` and `/newgame` completion contract waits on (`GameManagerService` resolves the completion `Task` on the first `UpdateTicked` *after* `SaveLoaded`), so the value is settled before the operation returns and before anything else can touch it.

**Why:** `AlwaysOn.HandleCaveChoice` set `caveChoice` from `OnOneSecondUpdateTicked`, guarded only by `!eventsSeen.Contains("65")` (no idempotency latch). `SaveImportTests.Import_SwapHost_CarriesMasterGatedState` seeds `caveChoice=1`; whenever the 12s automation pass fired *after* the seed but before `"65"` was present, it re-applied the server's mushroom default (`2`) over the seeded value → ~33% flake (`Expected 1, Actual 2`). The latency trap (the sibling rule) was already known; the *correctness/race* trap is distinct — same loop, different failure mode. The fix moved the choice into a `PreSeedChoices()` called from `OnSaveLoaded` (alongside the fishing/training-rod pre-seeds), which removed the timing dependence entirely.

**How to apply:** When adding or reviewing a host-side write that should happen once per load, ask "could another writer (a test seed, a command, the engine's own day-start) touch this same field in the window after load?" If yes, put it on the `OnSaveLoaded` pre-seed path so it's deterministic and settled before the completion contract resolves — don't park it on `OnOneSecondUpdateTicked` with only a mutable-game-state guard. Keep the single-shot signal a save-persisted one (`eventsSeen` membership, object existence) so a `/reload` re-run is a clean no-op; a re-run of the pre-seed is then harmless. Latency-sensitive *sequential* handlers still belong in `OnUpdateTicked` (per the sibling rule); this rule is specifically about *one-shot writes that something else can race*.
