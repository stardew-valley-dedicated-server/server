# E2E tests assert via the server HTTP API snapshot, not mod events

E2E tests assert against the **server HTTP API snapshot** — `/cabins` (`CabinsResponse`: TotalCount/AssignedCount/AvailableCount + per-cabin OwnerId/OwnerName/IsAssigned), `/farmhands` (`FarmhandsResponse`: per-slot Id/Name/IsCustomized), `/players`, plus client-side `/status` and `/wait/*` — and the `WaitFor*` helpers on `ServerApiClient`.

Mod `Diagnostics.ModEventLog.Emit(...)` events (`cabin_*`, `auth_*`, …) are transported via `Console.Out` lines prefixed with `SDVD_EVENT` → `SimpleContainerLogStreamer` → `infrastructure.jsonl` on disk. They are **diagnostics only**. No test asserts on a mod-emitted event, and there is **no "wait for mod event X" API** — the event stream is not a test assertion surface.

**Caveat — the snapshot can't see a stuck claim.** `/cabins` `IsAssigned` requires `owner.isCustomized == true` and `AvailableCount = TotalCount − AssignedCount`, so a cabin stuck with `userID` set but `isCustomized == false` (the abandoned-claim bug) still counts as **available** — indistinguishable from a healthy empty slot. Proving such a heal needs a **functional** gate (a real claim attempt succeeds, or exhaust all other slots to force a joiner onto the stuck one), not a snapshot count.

**Caveat — `/reload` and `/newgame` completion is gated on `SaveLoaded`, closing the migration-timing seam, but a second seam (published-snapshot staleness) is still open.** `ReloadServerAsync()` / `CreateNewGameOnServerAsync()` resolve only after `GameLoop.SaveLoaded` has fired — its handlers run the cabin migration/sync/sweep and `EnsureAtLeastXCabins`. This is by contract (`GameManagerService` resolves the completion `Task` on the first `UpdateTicked` *after* `SaveLoaded`, not when `LoadSave()` / `CreateNewGame()` returns — those only *arm* the loader; the world loads over later ticks). That guarantees the **game state** is final, but read-only endpoints serve a separately-republished `GameStateSnapshot` (1 Hz wall-clock, `ApiService.cs`) that is **not** force-refreshed at completion — so a snapshot captured mid-load, before the migration ran, can still be what a test's first post-reload `/cabins` read observes. **Don't add a settle-poll** for this (`retry-is-evidence-of-root-cause.md`) — see `.claude/plans/bugs/cabins-snapshot-stale-at-reload-completion.md` for the fix (republish the snapshot at completion) and its status. The bug: `MoveToStack_UnclaimedCabinSweptOnReload` flakes because the post-reload `/cabins` read can land against a snapshot captured *before* the `None→CabinStack` migration ran.

**Sub-caveat — `/newgame` SaveLoaded alone does NOT mark the saved world; its day-transition save runs later.** `CreateNewGame` arms `Game1.NewDay(0f)` (which only sets `newDay=true`; the transition + `SaveGame.Save` are *deferred*, pumped over later ticks by `newDayAfterFade`), then `setGameMode(3)` raises `SaveLoaded`. Those two race: `SaveLoaded` can fire at **day 0, pre-save**, so resolving `/newgame` there returns while the fresh game's own save is still in flight — a client that then joins/customizes races the save and loses its writes on the next reload (`AbandonedClaim_SweptOnReload`, 33% flake; the reload loaded the stale pre-customization world). Fixed by gating `_newGameCompletion` on **both** `SaveLoaded` *and* `ApiService.ComputeDayTransitionComplete()` (false until `newDaySync` finishes), so `/newgame` only completes once the new-day save is durable. `/reload` is exempt — it doesn't arm a fresh `NewDay`, so its `SaveLoaded` already marks the final world (the caveat above holds as-is for reload).

**How to apply:** Gate E2E assertions on `ServerApi.GetCabins/GetFarmhands/GetPlayers` + the `WaitFor*` helpers, never on an observed mod event (a plan once proposed asserting `cabin_claim_abandoned` — not achievable). When the snapshot can't distinguish the state you care about, design a functional probe instead.

**Give every assertion a description so a failure self-identifies the field.** A bare `Assert.Equal(1, state.MasterCaveChoice)` surfaces only `Assert.Equal() Failure: Expected 1, Actual 2` in `summary.json` — no field name, no expected behavior — which is dead weight when triaging a flake. Use the message-bearing overload (`Assert.True(cond, "<field> must <expected>; got <actual> (<what the wrong value means>)")`), matching the rest of the file's assertions; the one bare assertion in a class is the one that's hardest to attribute when it fails.
